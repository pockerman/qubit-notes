## Introduction

Before embarking on the journey of distributed system design let's first try to answer the question what is a distributed system.
There are plenty of definitions available. Below is one that is attributed to Leslie Lamport [1]:


_A distributed system is one in which the failure of a computer you didn't even know existed can render your own computer unsuable._

Another definition is, see [2],

_A distributed system is a collection of collaborating, concurrent components that communicate with one another by sending and receiving messages over a network_

Perhaps the most widedly known distributed system is the internet. Another example of a distributed system is, or can be, depending
on the needs, is a web application. Since this is a very common example of a distributed system, we will use it as the driving example in these notes.
Specifically, we will start with a set up where everything we need can be hosted on one server and then throughout these notes we will discuss challenges
we need to address and techniques we can use to solve these challenges in order to scale our system to millions of users.

Let's consider therefore the case of a web application. The example is taken from [3]. Initially, we will assume that the web application, the database and the cache layer are all running on the same machine.
The following figure schematically captures this assumption. The figure illustrates that there are two types of clients that our web server has to serve; a mobile application and a typical web application. Typically, these two applications have different requirements in terms of the responses and thus our web server needs to account for this.

Let's investigate at a very high level the flow of a request coming from a client.

1. A client accesses a website through domain names e.g. _mydomain.com_. The browser will send this domain name to a DNS service, see <a href="dist_systems/2025-04-18-domain-name-system.md">qubit-note: Distributed Systems Series |  Domain Name System</a>.
2. The DNS service will return the IP address to the browser (see Appendix below how to figure out the IP of your machine)
3 The browser will send HTTP requests to the web server sitting at the identified IP address
4. The web server returns HTML or JSON or XML or any other response that is configured to send

We can break the web application and the database to different servers ending in a configurartion where we have a web/mobile tier and database tier.

## Summary

A distributed system is a collection of independent components that run concurrently and communicate over a network to work together as one system, a concept famously illustrated by Leslie Lamport’s definition that a failure in an unknown computer can make your own unusable. The notes use a web application as a running example: initially, a single server may host the web app, database, and cache while clients (such as browsers or mobile apps) send requests after resolving a domain name through the Domain Name System. As user demand grows, this architecture must evolve by separating components into different tiers—such as a web tier and a database tier—so they can scale independently. However, distributing components across machines introduces new challenges, especially coordination and handling unreliable network communication, which distributed system design must address.

The next section will discuss the core challenges one has to face when designing a distributed system.

## Appendix

There are various methods to figure out the IP on a Linux machine depending on whether you want your local (private) IP or your public IP.


### Find Your Local IP Address 


```
ip addr
```

or shorter:

```bash
ip a
```

Look for:

- `inet` under your active interface (like `eth0`, `ens33`, `wlan0`)

Example:

  ```
  inet 192.168.1.25/24
  ```

If you want a cleaner output:

```bash
hostname -I
```


### Find Your Public IP Address (Internet IP)

```bash
curl ifconfig.me
```

or:

```bash
curl icanhazip.com
```

(Requires internet connection.)
